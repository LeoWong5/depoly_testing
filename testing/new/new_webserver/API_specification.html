The Front-Back API (Low-Level)

This is a full low-level specification of the Transport app API for communication between the front end and the back end. The equivalent high-level specification is on the design doc.

TLDR

  • The API communicates via HTTPS/3 which is secure and fast
  • The API should support HTTPS/1.1 if HTTPS/3 is not available on the client
  • The API is fully RESTful - no state is assumed serverside
  • The API is designed to enforce security and validation on requests
  • The front-end client will implement functionality to send requests and read response data
  • API calls use HTTP GET requests to pseudo-resources
  • Pseudo-resources use RFC 3989's [1] URI queries
  • This resource lookup style supports easy implementation of serverside and clientside caching
  • In particular, GET is used rather than POST to assist with caching
  • The back-end webserver will implement functionality to handle requests, query the database, marshall data, and return responses
  • Response message bodies are JSON text in formats specified below
  • All responses must include two fields: response-type and api-version
  • The client must ignore responses that do not meet the above requirement
  • Responses include HTTP status codes to indicate errors [2]
  • Other error data may also be included within returned JSON
  • Where an error status code is raised, the server should assume that the response body might be ignored by the client
Formatting note: in requests/responses, time should be formatted according to the following rules:
  1. Dates are strings in the format: "YYYY-MM-DDTHH:mm:ssZ", where 'T' indicates beginning of time descriptor and 'Z' indicates "relative to Zulu time" (aka UTC/GMT)
  2. Durations are described as JSON Numbers, in MINUTES
This format is a standard [3] and has been chosen due to Python and Javascript library support.

Note that all API requests may return a status code of 429 Too Many Requests. This is for DoS protection.

Note on server Exception Handling: when returning an error code, the server should still populate the body of the message such that the API is RESTful - this means, include the "response-type" and "api-version" properties as well as (potentially invalid) Node IDs or client-defined Datetimes in relevant fields. This means the client is aware what request specifically the error response is referring to.

Data Structure Format Notes
Response (General Format) 

{
	"api-version": VERSION as String,
	"response-type": TYPE as String,
	...  // response body 
}
All server responses must state the API version, currently "1.1.1" and the type of the response (i.e. which type of API request it corresponds to). Major version increases (the first digit) will be reserved for changes that are considered backwards-incompatible.
  • response-type is exactly one of: ["map", "timetable", "route", "service", "live-tracking"]
The version attribute has been omitted in the API below, but users should always include it.
Geo-Location 

{
	 // example GPS... 
	"lat": 54.054266697674485,
	"long": -2.8100538254190233
}
A GPS location where latitude/longitude are floats (generally, JSON Numbers). They should be as high-precision as possible where data is available. Used for stops, user location, etc.
Node 

{
	"id": e.g. "8712938",
	"type": e.g. "bus",
	"name": e.g. "Common Garden Street",
	"location": {... Geo-Location}
}
Nodes represent a individual transport stop/station of any kind, where:
  • "id" is a unique String ID, assigned by back-end. Consistent across all clients.
  • "type" is one of ["bus", "train", "tram"] - i.e. what vehicles stop here.
  • "name" is the displayed name of the stop.
  • "location" is the map GPS of the stop - see Geo-Location.
For performance purposes, the client will only request partial information on nodes initially, waiting until Timetable/Service/Tracking data is required for a specific node before requesting it (lazy loading). The client can then bundle this timetable data with its internal representations of Nodes.
Timetable 

{
	"arrivals": [
		{
			"time": DATETIME as String,
			"name": e.g. "1A",
			"operator": e.g. "Stagecoach",
			"status": VEHICLE STATUS as String
		},
		{
			...  // another arrival listing 
		}
	],
	"last-updated": DATETIME as String,
	"from-time": DATETIME as String,
	"node": NODE ID as String
}
Timetables have "arrivals" arrays containing a sequence of arrivals for a given node. In practice, they can only be transferred with the explicit context of the node (see API). Each array item is an object where:
  • "time" is the *expected* arrival time of the vehicle (which have been modified serverside if dynamic data for the vehicle is available). See tldr section for formatting note.
  • "name" is the service display name.
  • "operator" is the operator display name.
  • "status" is a String status code that provides info about the listing. It is one of: ["scheduled", "cancelled", "delayed", "on-time", "unknown"] . A scheduled listing may be on-time or delayed, but does not have available tracking data. The server should report cancelled listings to the client if they are upcoming (in *realtime*) but may choose to omit cancelled listings for timetable lookups several days in advance. The server should not report a vehicle as delayed unless it is expected to be late by more than a minute.
  • "from-time" is the date-time (possibly now) for which this timetable schedule starts at. For example, the user may wish to view the scheduled timetable three days in advance of a journey. This should match a datetime provided in a client request (see TimetableLookup).
  • "node" is the node ID for which this timetable is being returned.
The server can decide the total number and start timing of the listings it returns and should do so in response to API requests. The server must set the "last-updated" parameter to the current time when responding to a timetable request. This ensures the client can flush timetable data and replace it only when necessary.
Transport Metadata 

{
	"weather": {
		"weather": WEATHER as String,
		"location": { ... Geo-Location}
	}
	"messages": ["Return tickets are now 10% off", ...],
	"warnings": ["Roads are icy, take care."]
}
Transport Meta is general information relating to transport in Lancashire that users should know when they open the website, where:
  • "weather" includes exactly one String from ["unknown", "clear", "cloudy", "rain", "storm", "snow", "snowstorm"] to represent the current weather at the given Geo-Location. In practice, the Geo-Location would have been supplied by the user upon connecting.
  • "messages" is an array of display String information to prompt the user with. For usability this should be minimal.
  • "warnings" is an array of display String warnings relating to danger/general disruption/cancellations that may affect the general user's journey. Note that warnings do not have any context of the user or where they are going - these are county-wide notices.
Messages and warnings may be empty arrays. The client should support all stated weather options, but the server does not have to be able to send them all if data is not available or applicable.
Path 

{
	"from": NODE ID as String,
	"to": NODE ID as String,
	"path": [Array of Geo-Location objects]
}
Path objects describe the path road path between two Nodes. Paths are NOT a route; they are the graphical line that should be drawn (e.g. matching the curvature of the road) on the client-side map.
  • "from" is a valid node ID that represents where this path starts
  • "to" is a valid node ID that represents where this path ends
  • "path" is a list of Geo-Location objects (see above), including the start and end locations, that represent the vertices of the path.
The given Nodes should be adjacent - i.e. the graphical path should go via a direct transport link between the nodes.
Route 

{
	"duration": MINUTES as Number,
	"type": ROUTE TYPE as String Array,
	"travel": [
		{
			"departure": DATETIME as String,
			"duration": MINUTES as Number,
			"node": NODE ID as String,
			"type": TRANSPORT METHOD as String
		},
		{
			 // another leg of the journey (vehicle change) 
		}
	],
	"destination": NODE ID as String,
	"path": [... Array of Path objects]
}
Routes describe (one possible) found route between two given Nodes. They include the literal sequence of transport links, and the visual path that should be drawn on the map, as well as a classification for route type.
  • "duration" is the overall time in minutes to travel from source to destination via this route.
  • "type" is a classification of this route to indicate to the user, who may prefer different options. Currently supported types are ["fastest", "fewest-changes", "least-walking"]. May have one, zero, or more types.
  • "travel" is an array of objects representing each "leg" of the journey. Each entry is boarding a new scheduled vehicle. As such, intermediate stops should not be included in the sequence. Entries can also represent walking between two stops when necessary. There is no entry included for the destination.
    • "departure" is a String Datetime (see tldr formatting note for more info). This is when the vehicle is scheduled to leave, or in the case of walking, when the user is expected to start walking having exited the previous vehicle.
    • "duration" is how long in minutes this leg of the journey will take.
    • "node" is the String ID of the Node from which this leg of the journey starts.
    • "type" is a String representing the method of transport for this leg of the journey, one of ["bus", "train", "tram", "walking"]
  • "destination" is the String ID of the destination Node that this route leads to.
  • "path" is the visual graphical path that should be drawn to represent the route on the map. It is an array of Path objects (see above). It should be of length "travel".length with a separate path between each node.
Vehicle 

{
	"name": e.g. "1A",
	"location: {  ... Geo-Location  }
}
A live-tracked vehicle that appears on the map. "name" is a displayed String and "location" is a Geo-Location object matching to the vehicle's live position.

Service & Interface Request Response
MapService Requests a full set of all Nodes (stops) to display on the map, and extra information (e.g. weather).

GET request to

/map?lat=number&long=number
Where the lat/long parameters are high-precision floating point numbers that represent the user's current location (to get precise weather data / relevant warnings).

OR, (if client has nodes cached locally)

GET request to

/map?cached=true?lat=number?long=number
 Returns full set of stops/nodes (only surface level data, not timetables) with map metadata. Or, if cached=true is included in the request, only returns the "information" field (no nodes). 

{ 

	"response-type": "map", 
	"nodes": [... Array of Node objects],
	"information": {... Transport Metadata object}

}
Exception: user has requested all nodes 'too much' in past minute (e.g. 10+ times, i.e. a suspected DoS attack or faulty client). Server responds with status code 429.
Exception: latitude or longitude parameters are misformatted or invalid. Server responds with status code 414 or 400.
TimetableService TimetableLookup Requests the Timetable data for a given group of Nodes, which should incorporate dynamic data (e.g. cancellations or delays) when possible. Timetable data is requested from a specific datetime, (e.g. when the user intends to depart. This supports lookups of multiple Nodes at a time for efficiency (reducing the Number of overall requests).

GET request to

/timetable?time=DATETIME&nodes=ID1,ID2,ID3
Where the time parameter is a Datetime formatted as in TLDR section. The nodes parameter is a comma-separated list of Node IDs that timetable lookups are required for. In practice, the client should not request more than 10 Timetables in a single GET. 		Returns full set of Node timetables where possible. 

{
	"response-type": "timetable",
	"excluded": [ ... Array of Node IDs ],
	"timetables": { ... Object matching Node IDs to Timetable objects },
}

 
	// Example timetables field: 
	"timetables": {
		"12389": { Timetable object (see above) for node 12389 },
		"49283": { Timetable object (see above) for node 49283 },
		"88943": { Timetable object (see above) for node 88943 }
	}
The server should include ALL IDs specified in the Request in the Response; however, for Node IDs that were invalid, unauthorised, or otherwise rejected, it should place those IDs in the "excluded" array rather than returning a timetable for them. Even if all nodes are invalid, a success status code should be returned.

Exception: URI too long to process (too many nodes specified). Server responds with status code 414.
Exception: DATETIME invalid / bad formatting / unreasonable. Server responds with status code 400.
TimetableService ServiceLookup Requests data about the services that run through a particular Node. This includes graphical map paths for those services as well as names.

GET request to

/services/NODE ID
Where "NODE ID" is the desired Node. 		Returns data about services that run through a Node. For practical reasons, the server may choose to return no services for some Nodes (e.g. train stations). 

{ 
	"response-type": "service", 
	"node": "NODE ID" as String,
	"services": [
		{
			"service-name": e.g. "1A",
			"path": { ... Path object }
		},  // ... other services 
	]
}
  • path is a Path object (see above) for the overall road/rail route the service takes.
Exception: Invalid Node ID. Server responds with status code 404.
TimetableService LiveTracking Requests the map position of vehicles of a particular service going to/from a particular Node.

GET request to

/tracking/NODE ID
Where "NODE ID" is the desired Node. 		Returns a list of all Vehicles somewhat closely approaching the given Node, and any vehicles that have recently departed the Node. Since not all vehicles support tracking, there may be no Vehicles returned in many cases. The server determines which Vehicles are considered close - for example, those within 10 minutes may be considered close in a frequent-traffic bus stop, or those within 30 minutes at a rural stop. 

{
	"response-type": "live-tracking",
	"node": "NODE ID" as String,
	"vehicles": [ ... Array of Vehicle objects ]
}
"vehicles" is a list of all relevant Vehicles (which encapsulate tracking data, see above).

Exception: Invalid Node ID. Server responds with status code 404.
RouteService Request a selection of routes between a given source and destination node at a given date and time.

GET request to

/routes?source=NODEID&dest=NODEID&time=DATETIME
Where "NODEID"-s are valid IDs of the source and destination nodes, and "time" is a String Datetime for when the route scheduling should begin (see TLDR for Datetime specification). 		Return a selection of routes between the given source and destination. The routing options computed should, where possible, offer a diverse range of routes reflecting different qualities that align with different user preferences (shortest time, fewest changes, least walking). See Route in the Data Structures above for more info. The amount of routes returned is up to the server and may vary between the same requests (due to dynamic data and serverside performance). 

{ 
	"response-type": "route",
	"source": "NODE ID" as String,
	"destination": "NODE ID" as String,
	"routes": [... Array of relevant Route objects ]
}

Note that visual default selection of the “main” path is performed client side based on preferences given by the user. The server should aim to exclude paths that are strictly worse than others from its output, e.g. for Underpass to Lancaster Bus Station, now, do not include a 100 that arrives in 15 minutes if there is a 1A in 2 minutes.

Exception: Invalid Node ID for source or destination. Server responds with status code 404. Exception: Invalid or misformatted Datetime parameter. Server responds with status code 404. Exception: Unreasonable Datetime parameter value (e.g. 1970, or 2029). Server responds with status code 422. Exception: Not a single route from source to destination could not be found (in reasonable time frame). Server responds with status code 500.
