CSV_BUS Foreign Key Logic
=========================
Leo, 03 / 03 / 2026

Key rules:
  - "id" is like a local name from XML, duplicated across files — NEVER use as FK
  - "uid" is the true globally unique key — ALWAYS use uid or index for FK
  - Index-range FKs: parent stores (start_index, end_index), child rows in that
    uid range [start_index .. end_index] belong to the parent
  - Nullable sentinel: -1 means "no ref", "" (empty) means "no ref"
  - Nullable columns discovered: #11 days_of_week_uid, #12 bho_uid,
    #13 so_uid, #14 garage_uid, #28 route_location index, #30 SDO index,
    #31 VJ_link index

========================================================================
  4. Hierarchy / Dependency Graph
========================================================================

  operator  (root)
    ├── garage
    ├── service
    │     └── line
    ├── journey_pattern  → also refs route, journey_pattern_section, service
    └─ (via vehicle_journey)

  bus_stop_point  (root, refs NaPTAN stop_point by stop_point_ref)
    └── route_link  (from/to_bus_stop_point_uid)

  route  (root)
    └── route_section  (index range)
          └── route_link  (index range)
                └── route_location  (index range)

  journey_pattern_section  (parent = journey_pattern via JP_section_start/end_uid)
    └── journey_pattern_link  (index range, also refs route_link by uid)

  vehicle_journey  (refs operator, service, line, journey_pattern, garage)
    ├── days_of_week            (parent_VJ_uid)
    ├── special_days_operation  (index range, nullable)
    ├── bank_holiday_operation  (parent_VJ_uid)
    └── vehicle_journey_link    (index range, also refs journey_pattern_link by uid)

  serviced_organisation  (root)
    └── serviced_organisation_date_range  (organisation_uid)
    └── vehicle_journey  (serviced_organisation_uid, nullable -1)


========================================================================
  5. Cross-Check with db.py  (database/db.py)
========================================================================

  db.py uses AUTOINCREMENT integer PKs (OPE_UID, SER_UID, etc.)
  and proper relational FOREIGN KEY constraints.

  Key architectural difference:
    CSV uses index-range FKs  (parent stores start/end index)
    db.py INVERTS these into child→parent direct FKs
    (load_data.py must resolve index ranges when inserting)

  CSV #11: vehicle_journey.days_of_week_uid  → days_of_week

  db.py:   bank_holiday_operation.VJ_UID → Vehicle_Journey  (inverted)

  Either direction works; db.py chose child→parent for consistency.      ✅ inverted

  ── Summary ────────────────────────────────────────────────────────

  ALL 33 CSV FK relationships are accounted for in db.py.
  No missing FKs.  No contradictions.
  The only structural difference is index-range → child-stores-parent inversion,
  which is the correct relational normalisation.
